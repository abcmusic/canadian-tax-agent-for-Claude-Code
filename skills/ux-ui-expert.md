# UX/UI Expert Skill

## Overview

This skill provides comprehensive UX/UI design expertise, covering user research, interface design, usability evaluation, accessibility standards, and modern design practices. Use this skill when designing user experiences, evaluating interfaces, conducting user research, or implementing accessible and usable digital products.

## Core Competencies

### 1. UX Research Methods

#### Qualitative Research

**User Interviews**
```
Structure:
1. Introduction & Rapport Building (5 min)
   - Explain purpose and confidentiality
   - Set comfortable tone
   - Get consent for recording

2. Warm-up Questions (5-10 min)
   - General background
   - Context setting
   - Build trust

3. Main Interview (30-40 min)
   - Open-ended questions
   - Follow-up probes
   - Active listening

4. Wrap-up (5 min)
   - Any additional thoughts?
   - Thank participant
   - Next steps

Best Practices:
- Ask "why" and "how" questions
- Avoid leading questions
- Use silence effectively
- Take notes on non-verbal cues
- Record for later analysis
```

**Contextual Inquiry**
```
Approach:
1. Master-Apprentice Model
   - Observe users in natural environment
   - Ask questions as they work
   - Understand context and constraints

2. Four Principles:
   - Context: Work in user's environment
   - Partnership: Collaborate with users
   - Interpretation: Verify understanding
   - Focus: Maintain research goals

3. Data Collection:
   - Photographs of workspace
   - Videos of tasks
   - Artifacts (documents, tools)
   - Field notes

Analysis:
- Identify patterns and pain points
- Map workflows and dependencies
- Document workarounds
- Uncover unstated needs
```

**Focus Groups**
```
Planning:
- 6-10 participants per group
- 60-90 minute sessions
- Homogeneous groups (similar backgrounds)
- Professional moderator
- Observer/note-taker

Discussion Guide:
1. Introduction (5 min)
2. Warm-up (10 min)
3. Main topics (60 min)
4. Wrap-up (15 min)

Facilitation Tips:
- Encourage all voices
- Manage dominant participants
- Probe for depth
- Watch for group dynamics
- Capture verbatim quotes
```

#### Quantitative Research

**Surveys & Questionnaires**
```
Design Principles:
1. Clear objectives
2. Appropriate question types:
   - Multiple choice (standardized)
   - Likert scales (attitudes)
   - Open-ended (insights)
   - Ranking (priorities)

3. Survey Structure:
   - Introduction (purpose, time)
   - Screening questions
   - Main questions (logical flow)
   - Demographics
   - Thank you

4. Question Quality:
   - Simple language
   - One idea per question
   - Avoid bias
   - Balanced options
   - Clear instructions

Statistical Sampling:
- Sample size calculators
- Random sampling
- Stratified sampling
- Response rate targets (30%+)
```

**Analytics & Metrics**
```
Key UX Metrics:

Task Success Rate:
- Formula: (Successful tasks / Total tasks) × 100
- Target: >90% for critical tasks

Time on Task:
- Baseline establishment
- Track improvements
- Identify bottlenecks

Error Rate:
- Count of errors per task
- Types of errors
- Recovery time

System Usability Scale (SUS):
- 10-question survey
- Score: 0-100
- Benchmark: 68 = average

Net Promoter Score (NPS):
- "How likely to recommend?"
- Score: -100 to +100
- Promoters (9-10) - Detractors (0-6)

Analytics Tools:
- Google Analytics (behavior flow)
- Hotjar (heatmaps, recordings)
- Mixpanel (event tracking)
- Amplitude (user journeys)
```

**Usability Testing**
```
Test Setup:
1. Recruit 5-8 participants per round
2. Create realistic scenarios
3. Prepare test materials
4. Set up recording equipment
5. Brief facilitator and observers

Test Protocol:
1. Welcome & consent (5 min)
2. Background questions (5 min)
3. Task scenarios (40 min)
4. Post-test questionnaire (10 min)
5. Debrief interview (10 min)

Task Scenarios:
"You need to change your delivery address
for an upcoming order. Please show me how
you would do that."

Facilitation:
- Think-aloud protocol
- Minimal intervention
- Neutral probes ("What are you thinking?")
- Note critical incidents
- Observe non-verbal cues

Metrics to Capture:
- Task completion rate
- Time to complete
- Error count and type
- Satisfaction rating
- User quotes and insights
```

**A/B Testing**
```
Process:
1. Hypothesis Formation
   "Changing button color from blue to green
   will increase click-through rate by 15%"

2. Test Design:
   - Control (A): Current design
   - Variant (B): Modified design
   - Random assignment
   - Sufficient sample size

3. Success Metrics:
   - Primary (e.g., conversion rate)
   - Secondary (e.g., time on page)
   - Guard metrics (e.g., bounce rate)

4. Statistical Analysis:
   - Confidence level (95%)
   - Statistical significance (p < 0.05)
   - Practical significance (effect size)

5. Implementation:
   - Winner deployed
   - Document learnings
   - Plan next iteration

Tools:
- Google Optimize
- Optimizely
- VWO (Visual Website Optimizer)
- Adobe Target
```

### 3. UI Design Principles

#### Visual Hierarchy

**Typography Hierarchy**
```
Scale System (1.250 - Major Third):

Display:     48px / 3rem    (H1, Heroes)
Heading 1:   38px / 2.4rem  (Page titles)
Heading 2:   30px / 1.9rem  (Section headers)
Heading 3:   24px / 1.5rem  (Subsections)
Body Large:  19px / 1.2rem  (Emphasis)
Body:        16px / 1rem    (Default)
Body Small:  13px / 0.8rem  (Labels)
Caption:     10px / 0.65rem (Metadata)

Weight Scale:
- Light (300): Supporting text
- Regular (400): Body text
- Medium (500): Emphasis
- Semibold (600): Subheadings
- Bold (700): Headings

Line Height:
- Headings: 1.2-1.3
- Body text: 1.5-1.6
- Captions: 1.4

Letter Spacing:
- Tight (-0.02em): Large headings
- Normal (0): Body text
- Wide (0.05em): Uppercase labels
```

**Color & Contrast**
```
Color System:

Primary Palette:
- Primary:    #0066CC (Brand, CTAs)
- Secondary:  #6C757D (Supporting)
- Accent:     #FF6B35 (Highlights)

Neutrals:
- Gray 900:   #1A1A1A (Headings)
- Gray 700:   #4A4A4A (Body)
- Gray 500:   #9E9E9E (Disabled)
- Gray 300:   #D4D4D4 (Borders)
- Gray 100:   #F5F5F5 (Backgrounds)

Semantic Colors:
- Success:    #28A745
- Warning:    #FFC107
- Error:      #DC3545
- Info:       #17A2B8

WCAG Contrast Requirements:
- Normal text (< 18px): 4.5:1 minimum
- Large text (≥ 18px): 3:1 minimum
- Interactive elements: 3:1 minimum

Tools:
- WebAIM Contrast Checker
- Stark (Figma plugin)
- ColorSafe
```

**Spacing & Layout**
```
8-Point Grid System:

Base Unit: 8px

Scale:
4px  (0.5×)  - Tight spacing
8px  (1×)    - Small spacing
16px (2×)    - Medium spacing
24px (3×)    - Large spacing
32px (4×)    - XL spacing
48px (6×)    - XXL spacing
64px (8×)    - Section spacing

Component Spacing:
- Input padding: 12px 16px
- Button padding: 8px 24px
- Card padding: 24px
- Section margin: 48px

Layout Grid:
- 12-column grid
- 24px gutters
- 1440px max width
- Responsive breakpoints:
  • Mobile: 320-767px
  • Tablet: 768-1023px
  • Desktop: 1024-1439px
  • Wide: 1440px+
```

#### Gestalt Principles

**Proximity**
```
Group related elements together:

❌ Poor:
[Name]     [Email]
[Address]  [Phone]

✅ Good:
[Name]
[Email]

[Address]
[Phone]

Application:
- Form field grouping
- Navigation clusters
- Card content sections
```

**Similarity**
```
Similar elements perceived as group:

Consistent Patterns:
- Same color for all links
- Same icon style throughout
- Uniform button styling
- Matching heading hierarchy

Differentiation:
- Primary vs. secondary actions
- Active vs. inactive states
- Data vs. labels
```

**Continuity**
```
Eye follows continuous lines:

Examples:
- Breadcrumb trails: Home > Category > Product
- Step indicators: 1 ── 2 ── 3 ── 4
- Timeline visualizations
- Progress bars

Implementation:
- Guide user's path
- Create visual flow
- Connect related elements
```

**Closure**
```
Mind completes incomplete shapes:

Applications:
- Icon design (minimal shapes)
- Loading states (partial circles)
- Truncated text with ellipsis
- Skeleton screens

Example:
◠ ◡ ◠ ◡  (Perceived as waves)
```

**Figure-Ground**
```
Distinguish foreground from background:

Techniques:
- Drop shadows
- Overlays (dark/light)
- Blur effects
- Contrast differences

Modal Pattern:
┌─────────────────────┐
│ [Blurred Content]   │ ← Background (ground)
│   ┌─────────────┐   │
│   │   Modal     │   │ ← Foreground (figure)
│   └─────────────┘   │
└─────────────────────┘
```

**Validation Patterns**
```
Validation Timing:

On Blur (After Field Exit):
✅ Good for: Email, phone, username
❌ Avoid for: Password (too early)

On Submit:
✅ Good for: Final validation
❌ Avoid as only validation (frustrating)

Real-time (As User Types):
✅ Good for: Password strength, username availability
❌ Avoid for: Most fields (distracting)

Hybrid Approach:
1. Basic format validation on blur
2. Server validation on submit
3. Real-time for specific cases

Error Message Guidelines:
❌ "Invalid input"
✅ "Email must include @"

❌ "Error 422"
✅ "This username is already taken"

❌ "Field required"
✅ "Please enter your email address"
```

### 5. Usability Heuristics

#### Nielsen's 10 Heuristics

**1. Visibility of System Status**
```
Always inform users about what's happening:

Examples:
✅ Upload progress: "Uploading... 45%"
✅ Form submission: "Saving your changes..."
✅ Loading data: Skeleton screen
✅ Current location: Highlighted nav item
✅ Processing: Disabled button with spinner

❌ Silent operations
❌ No feedback after actions
❌ Unclear what's happening
```

**2. Match Between System and Real World**
```
Speak the user's language:

✅ "Trash" instead of "Delete permanently"
✅ "Shopping cart" instead of "Purchase container"
✅ Calendar icon for dates
✅ Familiar metaphors (folders, files)

❌ Technical jargon
❌ System-oriented terms
❌ Unclear icons
```

**3. User Control and Freedom**
```
Provide "emergency exits":

✅ Undo/Redo functionality
✅ Cancel buttons in modals
✅ Clear navigation path back
✅ Breadcrumbs
✅ "Are you sure?" confirmations

Implementation:
- Ctrl+Z / Cmd+Z for undo
- ESC to close modals
- Back button always works
- Exit without saving option
```

**4. Consistency and Standards**
```
Follow platform conventions:

Internal Consistency:
- Same button styles throughout
- Consistent terminology
- Uniform interaction patterns
- Predictable layouts

External Consistency:
- Platform guidelines (iOS, Android, Web)
- Industry standards
- Common UI patterns
- Established conventions

Example:
✅ "Sign In" or "Log In" (pick one, use everywhere)
❌ "Sign In" on homepage, "Log In" in app
```

**5. Error Prevention**
```
Prevent problems before they occur:

Confirmation Dialogs:
┌─────────────────────────┐
│ Delete 5 items?         │
│                         │
│ This cannot be undone.  │
│                         │
│ [Cancel] [Delete]       │
└─────────────────────────┘

Constraints:
- Date picker (can't select past dates)
- Number input (min/max values)
- Disabled invalid options
- Smart defaults

Helpful Hints:
- Password requirements shown upfront
- File type/size limits displayed
- Character counters before limit
```

**6. Recognition Rather Than Recall**
```
Minimize memory load:
```

### Example 5: Design System Component - Button

**Complete Button Component**
```jsx
// Button.jsx
import React from 'react';
import PropTypes from 'prop-types';
import './Button.css';

const Button = ({
  variant = 'primary',
  size = 'medium',
  disabled = false,
  loading = false,
  icon,
  iconPosition = 'left',
  children,
  onClick,
  type = 'button',
  ariaLabel,
  ...props
}) => {
  const classNames = [
    'button',
    `button--${variant}`,
    `button--${size}`,
    icon && !children && 'button--icon-only',
    loading && 'button--loading'
  ].filter(Boolean).join(' ');

  return (
    <button
      type={type}
      className={classNames}
      disabled={disabled || loading}
      onClick={onClick}
      aria-label={ariaLabel}
      aria-disabled={disabled || loading}
      {...props}
    >
      {loading && (
        <span className="button__spinner" aria-hidden="true">
          <svg viewBox="0 0 24 24">
            <circle cx="12" cy="12" r="10" />
          </svg>
        </span>
      )}

      {icon && iconPosition === 'left' && !loading && (
        <span className="button__icon button__icon--left">
          {icon}
        </span>
      )}

      {children && (
        <span className="button__text">
          {children}
        </span>
      )}

      {icon && iconPosition === 'right' && !loading && (
        <span className="button__icon button__icon--right">
          {icon}
        </span>
      )}
    </button>
  );
};

Button.propTypes = {
  variant: PropTypes.oneOf([
    'primary',
    'secondary',
    'tertiary',
    'danger',
    'success'
  ]),
  size: PropTypes.oneOf(['small', 'medium', 'large']),
  disabled: PropTypes.bool,
  loading: PropTypes.bool,
  icon: PropTypes.node,
  iconPosition: PropTypes.oneOf(['left', 'right']),
  children: PropTypes.node,
  onClick: PropTypes.func,
  type: PropTypes.oneOf(['button', 'submit', 'reset']),
  ariaLabel: PropTypes.string
};

export default Button;
```
